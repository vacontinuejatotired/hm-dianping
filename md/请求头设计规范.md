# 请求头设计规范

> 前后端对接的唯一参考。本文档定义认证相关的所有请求头/响应头规则。

***

## 1. 请求头定义

| 头名称 | 格式 | 必填 | 用途 |
|--------|------|------|------|
| `authorization` | `Bearer <access_token>` | 是 | 身份凭证（Access Token），每次请求都携带 |

**注意**：

- **强制全小写** `authorization`，前后端统一大小写，避免大小写不敏感问题
- `Refresh-Token` **不放在请求头**，而是由后端设为 **httpOnly Cookie**，浏览器自动携带，JS 不可读
- `GET /shop/**`、`GET /voucher/**`、`GET /blog/hot` 等公开接口可不传任何 Token
- **用户被删除**：数据库中的用户被删除后，已签发的 Token 在 Redis 中仍然有效直到 TTL 过期。当前设计不做主动吊销——这是一个已知限制，可接受（管理端删除用户后，Token 最多存活 30 分钟）

***

## 2. 响应头定义

| 头名称             | 出现时机              | 说明                                                                 |
| --------------- | ----------------- | ------------------------------------------------------------------ |
| `authorization` | 登录成功 / Token 刷新成功 | `Bearer <new_access_token>`。**强制全小写 `authorization`**，前端统一用 `response.headers.get('authorization')` 读取 |
| `Set-Cookie`    | 登录成功 / Token 刷新成功 | `refresh_token=<token>; HttpOnly; Secure; SameSite=None; Path=/` |

**说明**：

- 登录成功时**一定**返回两个凭证——access\_token 在响应头，refresh\_token 在 Set-Cookie
- 拦截器在 Token 续期/过期刷新时**才**返回新凭证，正常请求不返回
- `Refresh-Token` 存入 **httpOnly Cookie**，JS 无法读取，自动随请求发送，不存在泄露风险
- 前端**每次收到响应都应检查** `authorization` 头，有值则更新本地存储。
- 后端 `CorsConfig` 必须通过 `addExposedHeader("authorization")` 暴露该头，否则前端 `response.headers.get()` 读到 null

```typescript
const newAccess = response.headers.get('authorization');
if (newAccess) {
    localStorage.setItem('access_token', newAccess.replace('Bearer ', ''));
}
// Refresh-Token 由 Set-Cookie 自动管理，前端不需要任何操作
```

## 3. Token 生命周期

```
发验证码 / 登录
    │
    ▼
签发 TokenPair ─────────────────────────────────────────┐
│  access_token: 30min                                    │
│  refresh_token: 7天                                      │
│  version: 自增ID (每次登录/password change bump)            │
    │                                                        │
    ▼                                                        │
正常请求 (access_token 有效)                                   │
    │                                                        │
    ├─ authorization 头 → 拦截器校验                           │
    │   └─ Caffeine 快速拒绝 → Redis 版本校验 → 放行            │
    │   （Refresh-Token 由浏览器通过 Cookie 自动携带）            │
    │                                                        │
    ▼                                                        │
access_token 临期 (剩余 < 5~10min)                            │
    │                                                        │
    ├─ 拦截器自动刷新 (不 bump version, 分布式锁保护)           │
    │   └─ 响应头返回新 access_token (同版本)                    │
    │   └─ Set-Cookie 新 refresh_token (轮转, 分布式锁防竞态)     │
    │                                                        │
    ▼                                                        │
access_token 过期                                              │
    │                                                        │
    ├─ 拦截器自动刷新 (bump version + 换 refresh_token)          │
    │   └─ 响应头返回新 access_token + Set-Cookie 新 refresh_token │
    │                                                        │
    ▼                                                        │
refresh_token 过期 / 被盗用触发 Reuse Detection                │
    │                                                        │
    └─ 返回 401 → 前端清除 access_token → 跳登录页                │
                                                              │
修改密码                                                       │
    │                                                         │
    └─ bump version + 签发全新 TokenPair → 旧 Token 全部作废 ──┘
```

## 4. Token 刷新场景分析

### 4.1 可以刷新的情况

| 场景             | access\_token 状态             | refresh\_token 状态 | 刷新方式                         | 版本号      | 业务合理性                                                                                           |
| -------------- | ---------------------------- | ----------------- | ---------------------------- | -------- | ----------------------------------------------------------------------------------------------- |
| **临期刷新**       | 有效，剩余 < 5-10min              | 有效                | `refreshDeadlineTokenScript` | **不变**   | ✅ **安全**：access\_token 尚未过期，只是即将过期。不 bump version 避免并发请求因版本不一致被误杀。refresh\_token **轮转**，由**分布式锁**保护避免并发竞态（详见 §4.5）。 |
| **过期刷新**       | 已过期（JWT ExpiredJwtException） | 有效                | `refreshDeadTokenScript`     | **bump** | ✅ **预期行为**。过期刷新可能存在并发（多标签页同时过期），但 version bump 后后来的请求携带旧 version，Redis 拒绝。后到者持有一个已过期的 token，重新登录是合理的，不是 bug。无需锁。 |
| **手动换发**（修改密码） | 有效（但即将作废）                    | 有效（但即将作废）         | `LoginSetToken.lua`          | **bump** | ✅ **合理**：用户主动改密码 = 新身份，旧 token 理应全部作废。**响应头返回**新 `authorization`（Bearer）+ `Set-Cookie` 新 refresh_token |

### 4.2 不能刷新的情况

| 场景                                      | access\_token 状态 | refresh\_token 状态        | 结果             | 原因                                                                                                                                                                                                                                                                                                                                              |
| --------------------------------------- | ---------------- | ------------------------ | -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **refresh\_token 过期**                   | 已过期              | 已过期（Redis 中已无 key）       | **401**        | refresh\_token 的 Redis key 已自动过期（7 天 TTL），无法验证旧值。用户需重新登录。                                                                                                                                                                                                                                                                                       |
| **refresh\_token 不存在**                  | 已过期              | 请求未携带（Cookie 中无值）        | **401**        | 前端清掉了 Cookie 或首次访问。无法刷新。                                                                                                                                                                                                                                                                                                                        |
| **version 冲突**（被后续登录覆盖）                 | JWT 签名有效且未过期     | 有效（Redis 中仍有值，TTL 未到）    | **401**        | **场景**：用户在设备 A 登录（version=5）→ 在设备 B 同一账号登录（version bump 到 6）→ 设备 A 拿着旧 access\_token（JWT 本身未过期，还剩 20 分钟）和设备 B 登录前的旧 refresh\_token Cookie 发请求。**拦截器校验**：JWT 解码成功（签名正确、exp 未到）→ Caffeine 版本快速校验 HIT\_MATCH（本地缓存还是 version=5）→ 走 Redis 最终校验 → Redis 中 version=6 > token 中的 version=5 → 拒绝。**结论**：JWT 有效 ≠ 请求被放行。版本号是独立于 JWT 的作废机制，新登录使旧 token 失效。 |
| **Reuse Detection**（refresh\_token 被盗用） | 任意               | 同一 refresh\_token 被第二次使用 | **401 + 吊销全部** | Lua 脚本检测到 refresh\_token 已被消费（轮转后旧值作废），旧 refresh\_token 再次出现说明被盗用。**吊销动作**：bump version + DEL 该用户所有 token key + 删除 Redis 中的 refresh_token 记录，使任何持有旧 token 的请求立即失效。                                                                                                                                                                                                                                                                |
| **账户锁定**                                | 任意               | 任意                       | **401**        | 密码登录连续失败 10 次，账户被锁 30 分钟。refresh 本身不需要密码，但 refresh\_token 有效期间账户被锁仍可刷新（业务决策：锁定的是密码登录，不影响已有会话）。                                                                                                                                                                                                                                                  |
| **JWT 签名无效**                            | JWT 解析失败         | 任意                       | **401**        | Token 被篡改或不是本系统签发。拒绝任何操作。                                                                                                                                                                                                                                                                                                                       |

### 4.3 边界情况分析

| 场景                                          | 行为                                        | 分析                                                                                                                                                           |
| ------------------------------------------- | ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 两个并发请求同时触发临期刷新 | 第一个获锁的刷新成功；第二个拿不到锁，跳过刷新，直接放行 | ✅ **安全**。分布式锁保证同一用户同时只有一个刷新请求写 Redis。拿不到锁的请求不阻塞，跳过刷新。下次请求再试。详见 §4.5。 |
| 同一设备上临期刷新和过期刷新同时触发                          | **技术上不可能发生**                              | 同一请求中 JWT 要么已过期（抛 ExpiredJwtException），要么未过期（走临期判断），不可能同时满足两个条件。                                                                                             |
| 不同设备上一个临期和一个过期刷新几乎同时触发                      | 临期刷新成功，过期刷新失败（version 已被 bump）            | ✅ **合理**。过期刷新的请求携带的 version 已过时。但此时临期刷新已经换发了新 access\_token，过期请求用新 token 重试即可。                                                                               |
| 用户在 A 设备登录 → B 设备登录（bump version）→ A 设备发起请求 | A 的请求返回 401                               | ✅ **安全**。B 设备登录 bump 了 version，A 设备的旧 token 理应失效。                                                                                                            |
| refresh\_token 还在有效期内但用户已经重新登录了             | 旧 refresh\_token 失效（LoginSetToken.lua 覆盖） | ✅ **合理**。新登录产生了新 refresh\_token，旧的自然作废。                                                                                                                      |
| 修改密码时并发请求正在处理                               | 修改密码 bump version → 并发请求收到 401            | ✅ **合理**。密码已改，旧 token 不应继续有效。前端应在修改密码后用新 token 重试。                                                                                                           |

### 4.4 临期刷新是否应该 bump version？

| 方案               | 优点                                                                     | 缺点                                                                 |
| ---------------- | ---------------------------------------------------------------------- | ------------------------------------------------------------------ |
| **不 bump + 分布式锁**（当前方案） | 并发刷新安全——分布式锁保证同一用户同时只有一个刷新请求写 Redis。version 不变，拿不到锁的请求跳过刷新不阻塞 | 旧 access\_token（同 version）在剩余有效期内仍可使用。理论上旧 token 被泄露仍有风险，但窗口期仅剩几分钟 |
| **bump version** | 旧 token 立即作废，安全窗口归零                                                    | 并发刷新时第二个请求必 401（version 已变），用户体验差                                  |

**结论**：保持**不 bump**。临期刷新加**分布式锁**保护 refresh token 轮转。

### 4.5 并发刷新竞态 — 加分布式锁解决

用户提供的分析场景：

```
请求 A 与 B 同时携带旧 refresh_token RT_0 触发临期刷新。
Java 分别在 Lua 之外生成新 refresh_token: RT_A 和 RT_B。
B 的 Lua 先执行：校验通过 → 写入 RT_B，覆盖 RT_0
A 的 Lua 后执行：携带 RT_0 去比较 Redis 中的 RT_B → 不匹配 → 422 → A 刷新失败
```

**解决方案**：在拦截器层加分布式锁，锁内委托 AuthService 刷新。锁忙时不阻塞，跳过刷新。

```java
// ===== 拦截器层 =====
String lockKey = "lock:refresh:" + userId;
boolean locked = Boolean.TRUE.equals(stringRedisTemplate.opsForValue()
        .setIfAbsent(lockKey, "1", 3, TimeUnit.SECONDS));
if (!locked) {
    response.setHeader("X-Token-Refresh", "skipped");
    return true; // 跳过刷新，不写回 authorization，避免旧值覆盖新值
}
try {
    TokenPair newPair = authService.refreshTokenPair(token, refreshToken, userId, version, isExpired);
    if (newPair == null) {
        response.setHeader("X-Token-Refresh", "failed");
        response.setStatus(401);
        return false;
    }
    response.setHeader("X-Token-Refresh", "ok");
    response.setHeader("authorization", "Bearer " + newPair.getAccessToken());
    response.addCookie(createRefreshTokenCookie(newPair.getRefreshToken()));
    return true;
} finally {
    stringRedisTemplate.delete(lockKey);
}
```

```java
// ===== AuthService 层（锁内执行，无需重复校验）=====
String newToken = jwtUtil.generateToken(userId, 30L, ChronoUnit.MINUTES, oldVersion);
String newRefreshToken = UUID.randomUUID().toString().replace("-", "");
// 直接 SET 写入，不调 Lua（锁内已做并发防护）
redisTemplate.opsForValue().set(LOGIN_USER_KEY + userId, newToken, 30, TimeUnit.MINUTES);
redisTemplate.opsForValue().set(LOGIN_REFRESH_USER_KEY + userId, newRefreshToken, 7, TimeUnit.DAYS);
return new TokenPair(newToken, newRefreshToken, oldVersion);
```

**合理性**：
- 临期刷新频率极低（每 30 分钟一次），锁几乎不竞争
- 锁等待 100ms >> Lua 执行时间（< 10ms），几乎不会超时
- 双重检查避免锁内重复签名，浪费 CPU
- 收益：refresh token 每次刷新都轮转，泄露窗口归零

**决策表**：

| 刷新类型 | access_token | refresh_token | 版本号 | 并发保护 |
|---------|-------------|---------------|--------|---------|
| 临期刷新 | **换新**（同 version） | **换新**（轮转） | **不变** | **分布式锁**，拿不到锁则跳过 |
| 过期刷新 | **换新** | **换新**（轮转） | **bump** | 无需显式锁。过期刷新可能存在并发（多标签页同时过期），但 version bump 后第二个请求携带的旧 version 会被 Redis 拒绝，直接返回 401。用户重试即可——这是预期的安全行为，不是 bug |
| 修改密码 | **换新** | **换新** | **bump** | 无需锁（用户主动操作，单线程） |

### 4.6 四种方案横评

#### 先看清服务器的真正成本

当前每个经过拦截器的请求，固定开销：

```
请求 → Cookie 解析 → Authorization 头解析 → JWT 解码(CPU)
     → Caffeine 版本校验(内存)
     → Redis GET versionKey(网络!)  ← 这才是大头
     → Caffeine 用户信息加载(内存)
     → 放行
```

**每个请求都要一次 Redis GET**。这才是"西瓜"——不管用哪个刷新方案，这个成本都在。

以 1000 req/s、RTT 35ms 为例：

| 操作 | 单次成本 | 每秒调用次数 | Redis 时间占用/秒 |
|------|---------|------------|-----------------|
| Redis GET（版本校验） | 35ms | 1000 次 | **35,000ms** = 35 个并发连接持续饱和 |
| 临期刷新（有锁，2 次 GET + 2 次 SET） | 140ms | ~0.5 次 | 70ms — 可忽略 |
| 过期刷新（Lua，1 次往返） | 35ms | ~0.1 次 | 3.5ms — 可忽略 |

> **结论**：服务器的核心瓶颈在**每个请求必经的 Redis GET**（35 个并发连接被吃死），不在刷新路径上。高 RTT 环境下应优先优化版本校验的 Redis 调用频率（加长 Caffeine TTL、合并批量校验），而不是纠结刷新用几次 RTT。

| 操作 | 单次成本 | 频率 | 总成本 |
|------|---------|------|--------|
| JWT 解码（HS256） | ~0.01ms CPU | 每个请求 | 低 |
| Caffeine 查内存 | ~0.001ms | 每个请求 | 可忽略 |
| **Redis GET** | **~1ms 网络** | **每个请求** | **🔴 最高** |
| 临期刷新（含锁+Lua） | ~5ms | 每 30 分钟一次，稀疏 | 🟢 可忽略 |

> 所以**四个方案在服务器负载上的差异几乎可以忽略**——不管选哪个，每个请求都要先过拦截器，都要 JWT 解码 + Redis GET。真正的大头开销跟刷新方案无关。

#### 横评

| 维度 | A: 临期不轮换只延期 | B: 锁内生成 + 原子写入（当前方案） | C: 专用刷新接口+前端单例 | D: 长有效期+滑动续期 |
|------|-------------------|-------------------------------|----------------------|------------------|
| **服务器负载** | 🔴 每个请求 Redis GET，和 B 一样 | 🔴 每个请求 Redis GET，和 A 一样 | 🔴 每个请求 Redis GET，和 A 一样，**还多一次前端转发的刷新请求** | 🔴 滑动续期需在每个请求上重签 JWT（同 B 的 CPU 开销）或维护 Redis 映射（引入有状态），实际并不省 |
| **实现复杂度** | ⭐ 最低。Lua 不改，只需加锁 | ⭐⭐ 低。Lua 加 SET 逻辑 | ⭐⭐⭐⭐ 高。新增接口+前端排队+后端校验 | ⭐⭐⭐ 看起来简单，但 JWT 签发后不能改 payload，滑动续期要么重新签发（同 B），要么维护 Redis 映射（引入状态） |
| **并发安全** | ✅ 无竞态（不写新值） | ✅ 锁内原子，已解决 | ✅ 前端单例保证同时只有一个刷新请求 | ✅ 无刷新，无竞态 |
| **RT 泄露风险** | ❌ 不轮转，泄露后可用到 7 天后过期 | ✅ 每次刷新都轮转，泄露窗口窄 | ✅ 每次刷新都轮转 | N/A（无 RT） |
| **前端复杂度** | ⭐ 无感 | ⭐ 无感 | ⭐⭐⭐ 需实现请求队列/单例模式 | ⭐ 无感 |
| **向后兼容** | ✅ 当前代码就是如此 | ⚠️ 需要改 Lua（加 SET） | ❌ 全新架构，前端后端都要大改 | ❌ 需要所有客户端更新 token 有效期 |
| **推荐度** | ⭐⭐⭐ | ⭐⭐⭐⭐（已选） | ⭐⭐ | ⭐⭐ |

#### 分析

- **方案 B 仍是当前最优选**。因为服务器的瓶颈根本不在刷新这块（30 分钟一次 vs 每秒数千请求），加分布式锁的开销稀释到可忽略。
- **方案 D 的滑动续期没有省掉任何事**——JWT 签发后不能修改 payload，你想"延长过期时间"就只能重新签发（等于又走一次 B 的流程），或者改成 Redis 维护 token→userId 映射（自废 JWT 无状态优势）。两头不讨好。
- **如果真要优化服务器**，应该砍的是**每个请求的 Redis GET**，而不是在 refresh token 方案之间纠结。那把 Caffeine 的 TTL 加长、减少穿透到 Redis 的频率，才是动到了"西瓜"。但这是另一个话题了。

**最终排序**：B > A > D > C

## 5. 安全设计决策

| 决策                              | 理由                                                 |
| ------------------------------- | -------------------------------------------------- |
| Access Token 存 localStorage     | 纯前端 SPA，无后端渲染，无法用 httpOnly Cookie 给 JS 用           |
| Refresh Token 存 httpOnly Cookie | **关键安全设计**：JS 不可读，XSS 无法窃取。浏览器自动携带，前端零操作           |
| Cookie Path 限制为 `/` | 保证所有后端接口都能收到 Refresh-Token Cookie |
| Cookie SameSite=None; Secure | 允许跨域携带 Cookie。前端需 `withCredentials: true`。生产环境 HTTPS 必须。**开发环境**没有 HTTPS，需降级为 `SameSite=Lax`（去掉 Secure），否则 Cookie 不生效 |
| 拦截器透明刷新                         | 不暴露 `/auth/refresh` 端点，避免被无限调用                     |
| Refresh Token 轮转                | 所有刷新场景都换发新 Refresh Token。临期刷新由**分布式锁**保护避免并发竞态（详见 §4.5）                 |
| Reuse Detection | 检测到同一 Refresh Token 被用两次 → 吊销全部 Token，暴露攻击。**注意**：网络重传可能导致误判。实现时用 Redis SETNX 1s TTL 做防抖窗口：`SET lock:reuse:{rt} 1 EX 1 NX`，若返回 0（1 秒内重复）则只记日志不吊销 |
| 临期刷新不 bump version              | 避免并发请求时第二个请求因版本不匹配被误判 401。风险窗口仅剩几分钟，可接受。详细分析见 §4.4 |
| 过期刷新 bump version               | 过期 Token 需要全新身份，旧 Token 理应全部作废                     |
| `authorization: Bearer` 格式（全小写） | 前后端统一大小写，避免大小写敏感问题。后端同时兼容 `Authorization`（大写）读取                                          |
| CORS 暴露 `authorization` 响应头 | 后端 `CorsConfig` 必须 `addExposedHeader("authorization")`，否则前端 `response.headers.get()` 始终为 null |

***

## 6. 前端实现要点

```typescript
// ============ 1. 请求拦截器 ============
// 全局配置：跨域携带 Cookie（SameSite=None 需要 withCredentials）
axios.defaults.withCredentials = true;

axios.interceptors.request.use(config => {
    const accessToken = localStorage.getItem('access_token');
    if (accessToken) {
        config.headers['authorization'] = `Bearer ${accessToken}`;
    }
    // Refresh-Token 由浏览器通过 httpOnly Cookie 自动携带，前端无需处理
    return config;
});

// ============ 2. 响应拦截器 ============
axios.interceptors.response.use(
    response => {
        // 更新 Access Token（拦截器在刷新成功时会返回新 token）
        const newAccess = response.headers.get('authorization');
        if (newAccess) {
            localStorage.setItem('access_token', newAccess.replace('Bearer ', ''));
        }
        // Refresh-Token 由 Set-Cookie 自动管理，前端不需要读取或存储
        return response;
    },
    error => {
        if (error.response?.status === 401) {
            // 🔴 401 原因无法区分（刷新冲突/Token 过期/账户锁定/签名无效），统一清登录
            // 只清 token 相关 key，避免误清其他业务数据
            localStorage.removeItem('access_token');
            // 注意：refresh_token 在 httpOnly Cookie 中，由后端 Set-Cookie 覆盖或过期
            // 前端无法主动清除 httpOnly Cookie，但后端返回 401 时不会再下发新 Cookie
            window.location.href = '/login';
        }
        return Promise.reject(error);
    }
);

// ============ 3. 登录后处理 ============
async function login(phone, code) {
    const response = await fetch('/user/login', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ phone, code })
    });
    // Access Token 从响应头取（全小写 authorization）
    const accessToken = response.headers.get('authorization')?.replace('Bearer ', '');
    if (accessToken) {
        localStorage.setItem('access_token', accessToken);
    }
    // Refresh Token 已通过 Set-Cookie 存入 httpOnly Cookie，无需前端处理
    return response.json();
}

// ============ 4. 公开接口列表（不传 Token） ============
const PUBLIC_PATHS = [
    '/user/code',
    '/user/login',
    '/shop/',
    '/shop-type/list',
    '/voucher/',
    '/blog/hot',
];

function isPublic(url) {
    return PUBLIC_PATHS.some(path => url.startsWith(path));
}
```

```java
@Value("${app.cookie.same-site}")
private String sameSite;
@Value("${app.cookie.secure}")
private boolean secure;

private Cookie createRefreshTokenCookie(String token) {
    Cookie cookie = new Cookie("refresh_token", token);
    cookie.setHttpOnly(true);
    cookie.setSecure(secure);
    cookie.setPath("/");
    cookie.setMaxAge(7 * 24 * 60 * 60);
    // SameSite 属性需手动加（Servlet API 不支持直接设置）
    response.addHeader("Set-Cookie", String.format(
        "refresh_token=%s; HttpOnly; %sSameSite=%s; Path=/; MaxAge=%d",
        token, secure ? "Secure; " : "", sameSite, 7 * 24 * 60 * 60
    ));
}
```

## 8. 方法契约

### 8.1 类型约定

| 值 | Redis 存储类型 | Java 类型 | 说明 |
|----|-------------|----------|------|
| version | **String**（`SET` 时 `String.valueOf()`） | **Long**（Redis GET 后 `Long.parseLong()`） | 拦截器比较时转 Long，确保数值比较而非字符串比较 |
| access_token | String | String | JWT 字符串 |
| refresh_token | String | String | UUID |
| userId | String | Long | Redis key 后缀 |

### 8.2 `generateTokenPair(userId, version)`

```
输入: userId (Long), version (Long, 可选)
输出: TokenPair { accessToken (String), refreshToken (String), version (Long) }
行为:
  1. 若 version 为 null → 调用 redisIdWorker.nextVersion(userId) 自增获取
  2. 若 version 不为 null → 使用传入的 version（临期刷新场景，不 bump）
  3. 生成 accessToken: JWT, 含 userId+version, 有效期 30min
  4. 生成 refreshToken: UUID, 不含任何身份信息
  5. 不写 Redis, 不设 Cookie（调用方负责）
```

### 8.3 `createRefreshTokenCookie(value)`

```
输入: refreshToken 字符串
输出: jakarta.servlet.http.Cookie
属性:
  - name: "refresh_token"
  - value: refreshToken
  - HttpOnly: true
  - Secure: true（生产）/ false（开发）
  - SameSite: None（生产）/ Lax（开发）
  - Path: "/"
  - MaxAge: 604800（7 天）
```

### 8.4 `jwtUtil.generateToken(userId, TTL, unit, version)`

```
输入: userId (Long), TTL (Long), unit (ChronoUnit), version (Long, 可为 null)
输出: String (JWT)
行为:
  1. claims: { userId, version(if not null) }
  2. 签名算法: HS256（对称密钥，在 JwtUtil 初始化时加载）
  3. 过期时间: now + TTL × unit
  4. 签发时间: now
  5. 返回紧凑格式 JWT 字符串
```
