# 请求头设计规范

> 前后端对接的唯一参考。本文档定义认证相关的所有请求头/响应头规则。

---

## 1. 请求头定义

| 头名称 | 格式 | 必填 | 用途 |
|--------|------|------|------|
| `Authorization` | `Bearer <access_token>` | 是 | 身份凭证（Access Token），每次请求都携带 |

**注意**：
- 使用标准 `Authorization: Bearer` 格式（非全小写 `authorization`）
- `Refresh-Token` **不放在请求头**，而是由后端设为 **httpOnly Cookie**，浏览器自动携带，JS 不可读
- `GET /shop/**`、`GET /voucher/**`、`GET /blog/hot` 等公开接口可不传任何 Token

---

## 2. 响应头定义

| 头名称 | 出现时机 | 说明 |
|--------|---------|------|
| `Authorization` | 登录成功 / Token 刷新成功 | `Bearer <new_access_token>` |
| `Set-Cookie` | 登录成功 / Token 刷新成功 | `refresh_token=<token>; HttpOnly; Secure; SameSite=Strict; Path=/` |

**说明**：
- 登录成功时**一定**返回两个凭证——access_token 在响应头，refresh_token 在 Set-Cookie
- 拦截器在 Token 续期/过期刷新时**才**返回新凭证，正常请求不返回
- `Refresh-Token` 存入 **httpOnly Cookie**，JS 无法读取，自动随请求发送，不存在泄露风险
- 前端**每次收到响应都应检查** `Authorization` 头，有值则更新本地存储：

```typescript
const newAccess = response.headers.get('Authorization');
if (newAccess) {
    localStorage.setItem('access_token', newAccess.replace('Bearer ', ''));
}
// Refresh-Token 由 Set-Cookie 自动管理，前端不需要任何操作
```

## 3. Token 生命周期

```
发验证码 / 登录
    │
    ▼
签发 TokenPair ─────────────────────────────────────────┐
│  access_token: 30min                                    │
│  refresh_token: 7天                                      │
│  version: 自增ID (每次登录/password change bump)            │
    │                                                        │
    ▼                                                        │
正常请求 (access_token 有效)                                   │
    │                                                        │
    ├─ Authorization 头 → 拦截器校验                           │
    │   └─ Caffeine 快速拒绝 → Redis 版本校验 → 放行            │
    │   （Refresh-Token 由浏览器通过 Cookie 自动携带）            │
    │                                                        │
    ▼                                                        │
access_token 临期 (剩余 < 5~10min)                            │
    │                                                        │
    ├─ 拦截器自动刷新 (不 bump version, 仅延长 TTL)              │
    │   └─ 响应头返回新 access_token (同版本)                    │
    │   └─ Set-Cookie 新 refresh_token (轮转, 旧值作废)           │
    │                                                        │
    ▼                                                        │
access_token 过期                                              │
    │                                                        │
    ├─ 拦截器自动刷新 (bump version + 换 refresh_token)          │
    │   └─ 响应头返回新 access_token + Set-Cookie 新 refresh_token │
    │                                                        │
    ▼                                                        │
refresh_token 过期 / 被盗用触发 Reuse Detection                │
    │                                                        │
    └─ 返回 401 → 前端清 Token → 跳登录页                       │
                                                              │
修改密码                                                       │
    │                                                         │
    └─ bump version + 签发全新 TokenPair → 旧 Token 全部作废 ──┘
```

## 4. Token 刷新场景分析

### 4.1 可以刷新的情况

| 场景 | access_token 状态 | refresh_token 状态 | 刷新方式 | 版本号 | 业务合理性 |
|------|------------------|-------------------|---------|--------|-----------|
| **临期刷新** | 有效，剩余 < 5-10min | 有效 | `refreshDeadlineTokenScript` | **不变** | ✅ **安全**：access_token 尚未过期，只是即将过期。不 bump version 避免并发请求因版本不一致被误杀。换发新 refresh_token（轮转）限制泄露影响。 |
| **过期刷新** | 已过期（JWT ExpiredJwtException） | 有效 | `refreshDeadTokenScript` | **bump** | ✅ **安全**：旧 access_token 已过期，bump version 让同版本号的其他 token（如果有）也失效。符合"过期即废弃"原则。 |
| **手动换发**（修改密码） | 有效（但即将作废） | 有效（但即将作废） | `LoginSetToken.lua` | **bump** | ✅ **合理**：用户主动改密码 = 新身份，旧 token 理应全部作废。新 refresh_token 通过 Set-Cookie 下发。 |

### 4.2 不能刷新的情况

| 场景 | access_token 状态 | refresh_token 状态 | 结果 | 原因 |
|------|------------------|-------------------|------|------|
| **refresh_token 过期** | 已过期 | 已过期（Redis 中已无 key） | **401** | refresh_token 的 Redis key 已自动过期（7 天 TTL），无法验证旧值。用户需重新登录。 |
| **refresh_token 不存在** | 已过期 | 请求未携带（Cookie 中无值） | **401** | 前端清掉了 Cookie 或首次访问。无法刷新。 |
| **version 冲突**（被后续登录覆盖） | 有效 | 有效（但 version 已过时） | **401** | 用户在其他设备登录产生了新 version，旧 token 的 version 小于 Redis 中的最新版本。**符合安全预期**：新登录使旧 token 作废。 |
| **Reuse Detection**（refresh_token 被盗用） | 任意 | 同一 refresh_token 被第二次使用 | **401 + 吊销全部** | Lua 脚本检测到 refresh_token 已被消费（轮转后旧值作废），旧 refresh_token 再次出现说明被盗用，吊销该用户所有 token。 |
| **账户锁定** | 任意 | 任意 | **401** | 密码登录连续失败 10 次，账户被锁 30 分钟。refresh 本身不需要密码，但 refresh_token 有效期间账户被锁仍可刷新（业务决策：锁定的是密码登录，不影响已有会话）。 |
| **JWT 签名无效** | JWT 解析失败 | 任意 | **401** | Token 被篡改或不是本系统签发。拒绝任何操作。 |

### 4.3 边界情况分析

| 场景 | 行为 | 分析 |
|------|------|------|
| 两个并发请求同时触发临期刷新 | 两个都刷新成功（version 不变，Lua 原子操作） | ✅ **安全**。临期刷新不 bump version，两个请求的 Lua 脚本都只校验旧 refresh_token + 旧 version。不会相互阻塞。 |
| 一个临期刷新和一个过期刷新并发 | 临期刷新成功，过期刷新失败（version 已被 bump） | ✅ **合理**。过期刷新的请求携带的 version 已过时。但此时临期刷新已经换发了新 access_token，过期请求用新 token 重试即可。 |
| 用户在 A 设备登录 → B 设备登录（bump version）→ A 设备发起请求 | A 的请求返回 401 | ✅ **安全**。B 设备登录 bump 了 version，A 设备的旧 token 理应失效。 |
| refresh_token 还在有效期内但用户已经重新登录了 | 旧 refresh_token 失效（LoginSetToken.lua 覆盖） | ✅ **合理**。新登录产生了新 refresh_token，旧的自然作废。 |
| 修改密码时并发请求正在处理 | 修改密码 bump version → 并发请求收到 401 | ✅ **合理**。密码已改，旧 token 不应继续有效。前端应在修改密码后用新 token 重试。 |

## 5. 安全设计决策

| 决策 | 理由 |
|------|------|
| Access Token 存 localStorage | 纯前端 SPA，无后端渲染，无法用 httpOnly Cookie 给 JS 用 |
| Refresh Token 存 httpOnly Cookie | **关键安全设计**：JS 不可读，XSS 无法窃取。浏览器自动携带，前端零操作 |
| Cookie Path 限制为 `/` | 保证所有后端接口都能收到 Refresh-Token Cookie |
| 拦截器透明刷新 | 不暴露 `/auth/refresh` 端点，避免被无限调用 |
| Refresh Token 轮转 | 每次刷新都换发新 Refresh Token，旧值作废，限制泄露影响 |
| Reuse Detection | 检测到同一 Refresh Token 被用两次 → 吊销全部 Token，暴露攻击 |
| 临期刷新不 bump version | 避免并发请求时第二个请求因版本不匹配被误判 401 |
| 过期刷新 bump version | 过期 Token 需要全新身份，旧 Token 理应全部作废 |
| `Authorization: Bearer` 格式 | 符合 HTTP 标准规范，兼容所有标准 HTTP 客户端 |

## 6. 前端实现要点

```typescript
// ============ 1. 请求拦截器 ============
axios.interceptors.request.use(config => {
    const accessToken = localStorage.getItem('access_token');
    if (accessToken) {
        config.headers['Authorization'] = `Bearer ${accessToken}`;
    }
    // Refresh-Token 由浏览器通过 httpOnly Cookie 自动携带，前端无需处理
    return config;
});

// ============ 2. 响应拦截器 ============
axios.interceptors.response.use(
    response => {
        // 更新 Access Token（拦截器在刷新成功时会返回新 token）
        const newAccess = response.headers['authorization'] || response.headers['Authorization'];
        if (newAccess) {
            localStorage.setItem('access_token', newAccess.replace('Bearer ', ''));
        }
        // Refresh-Token 由 Set-Cookie 自动管理，前端不需要读取或存储
        return response;
    },
    error => {
        if (error.response?.status === 401) {
            localStorage.clear();
            window.location.href = '/login';
        }
        return Promise.reject(error);
    }
);

// ============ 3. 登录后处理 ============
async function login(phone, code) {
    const response = await fetch('/user/login', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ phone, code })
    });
    // Access Token 从响应头取
    const accessToken = response.headers.get('Authorization')?.replace('Bearer ', '');
    if (accessToken) {
        localStorage.setItem('access_token', accessToken);
    }
    // Refresh Token 已通过 Set-Cookie 存入 httpOnly Cookie，无需前端处理
    return response.json();
}

// ============ 4. 公开接口列表（不传 Token） ============
const PUBLIC_PATHS = [
    '/user/code',
    '/user/login',
    '/shop/',
    '/shop-type/list',
    '/voucher/',
    '/blog/hot',
];

function isPublic(url) {
    return PUBLIC_PATHS.some(path => url.startsWith(path));
}
```
