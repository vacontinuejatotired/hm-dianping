## 1. 项目概述

**黑马点评**是一个类大众点评的本地生活服务平台，核心功能包括：

- 用户注册/登录（手机验证码 + 密码）
- 商铺浏览（按类型、按名称、按地理位置）
- 探店笔记（发布、点赞、查看点赞用户、关注者动态流）
- 关注/取关/共同关注
- 优惠券查看与秒杀抢购
- 用户签到

### 技术栈

| 组件 | 技术 |
|------|------|
| 后端框架 | Spring Boot 3.4.4 |
| 数据库 | MySQL (MyBatis-Plus) |
| 缓存 | Redis + Caffeine 本地缓存 |
| 消息队列 | RabbitMQ |
| 认证 | JWT 双Token（Access Token + Refresh Token） |
| ID生成 | 基于 Redis 的雪花算法 |
| 跨域 | CorsFilter 全局配置（开发期允许所有来源） |

## 3. 统一响应格式

所有接口返回统一的 `Result` 对象：

```json
{
  "success": true,
  "errorMsg": null,
  "data": { ... },
  "total": null
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `success` | Boolean | 请求是否成功 |
| `errorMsg` | String | 失败时的错误消息 |
| `data` | Object | 响应数据（可能为List、Object或null） |
| `total` | Long | 分页场景下的总数（当前未使用） |

### 成功示例

```json
// GET /shop-type/list
{
  "success": true,
  "errorMsg": null,
  "data": [
    {
      "id": 1,
      "name": "美食",
      "icon": "/types/1.png",
      "sort": 1
    }
  ],
  "total": null
}
```

### 失败示例

```json
// POST /user/login 验证码错误
{
  "success": false,
  "errorMsg": "验证码错误",
  "data": null,
  "total": null
}
```

### 全局异常

未捕获的 RuntimeException 统一返回：

```json
{
  "success": false,
  "errorMsg": "服务器异常",
  "data": null,
  "total": null
}
```

## 4. 认证机制（双Token）

### 4.1 概述

项目使用 **Access Token + Refresh Token 双Token** 机制：

| Token | 有效期 | 作用 |
|-------|--------|------|
| `authorization`（Access Token） | 30 分钟 | 携带在请求头中，证明用户身份。**强制全小写** |
| `Refresh-Token` | 7 天 | 当 Access Token 过期时，用于刷新新 Token |

### 4.2 登录流程

```
前端                                   后端
  |                                     |
  |  1. POST /user/code?phone=xxx       |  发送验证码
  |------------------------------------->|
  |                                     |
  |  2. POST /user/login                |  登录（验证码或密码）
  |  { phone, code } 或 { phone, password }  |
  |<-------------------------------------|
  |  响应头：                            |
  |  authorization: Bearer <Access Token>|  ← 全小写
  |  Set-Cookie: refresh_token=<...>    |  ← httpOnly，JS 不可读
  |   （HttpOnly; Secure; SameSite=None）|
  |                                     |
  |  3. 前端保存 Access Token，Refresh   |
  |     Token 由浏览器 Cookie 自动管理   |
```

**密码登录说明**：
- `code` 和 `password` 二选一，同时传时优先使用 `code`
- 密码要求：至少 8 位，含大写字母 + 小写字母 + 数字
- 密码错误超过 10 次 → 账户锁定 30 分钟，返回 `"账户已锁定，请 30 分钟后重试"`
- 输入错误提示统一为 `"账号或密码错误"`（不区分用户是否存在，防止枚举攻击）

### 4.3 请求认证

登录后的所有需要登录的接口，请求头中必须携带：

```
authorization: Bearer <Access Token>
```

> **注意**：
> - **强制全小写** `authorization`，前后端统一
> - `Refresh-Token` 不放在请求头，由浏览器通过 httpOnly Cookie 自动携带
> - 响应头中**也会返回新的** `authorization`，前端每次请求后都需要用响应头中的值更新本地存储的 Access Token（因为拦截器可能会自动续期）
> - 跨域请求需配置 `withCredentials: true`，否则 Cookie 不会被发送

### 4.4 Token 自动续期机制

后端有一个 `RefreshTokenInterceptor` 拦截器，会在每次请求时做以下事情：

1. **校验 Access Token** 是否有效
2. **已过期** → 从 Cookie 读取 Refresh Token 尝试刷新（原子操作，Lua 脚本保证）
3. **刷新成功** → 响应头返回新的 `authorization: Bearer <token>` + `Set-Cookie` 更新 Refresh Token
4. **锁被占用**（并发刷新）→ 跳过刷新，响应头带 `X-Token-Refresh: skipped`，前端保持现有 Token
5. **两者都无效** → 返回 `401 Unauthorized` + `X-Token-Refresh: failed`，前端应跳转到登录页

> **响应头 `X-Token-Refresh`** 用于标识本次请求的 Token 刷新状态：
> - `ok` — 刷新成功，`authorization` 头中有新 Token
> - `skipped` — 刷新跳过（并发），无新 Token，前端保持现有 Token
> - `failed` — 刷新失败，伴随 401

### 4.5 前端 Token 管理最佳实践

```typescript
// 1. 全局配置：跨域携带 Cookie（SameSite=None 需要 withCredentials）
axios.defaults.withCredentials = true;

// 2. 请求拦截器
axios.interceptors.request.use(config => {
    const accessToken = localStorage.getItem('access_token');
    if (accessToken) {
        config.headers['authorization'] = `Bearer ${accessToken}`;
    }
    // Refresh-Token 由浏览器通过 httpOnly Cookie 自动携带，前端无需处理
    return config;
});

// 3. 响应拦截器
axios.interceptors.response.use(
    response => {
        const newAccess = response.headers['authorization'];  // Axios headers 是对象，不能用 .get()
        if (newAccess) {
            localStorage.setItem('access_token', newAccess.replace('Bearer ', ''));
        }
        // Refresh-Token 由 Set-Cookie 自动管理
        return response;
    },
    error => {
        if (error.response?.status === 401) {
            localStorage.removeItem('access_token');
            window.location.href = '/login';
        }
        return Promise.reject(error);
    }
);
```

## 5. API 接口列表

### 5.1 用户模块

**Base URL**: `/user`

#### 5.1.1 发送手机验证码

```
POST /user/code?phone={phone}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `phone` | String | 是 | 手机号码 |

**响应**: `Result`（data 为验证码，仅开发环境返回，生产环境 data 为 null）

```json
// 开发环境响应示例
{
  "success": true,
  "data": "123456"
}
```

> **注意**：开发环境直接返回验证码数字，方便调试。生产环境 data 为 null（验证码仅通过短信发送）。

#### 5.1.2 登录

```
POST /user/login
Content-Type: application/json

{
  "phone": "13800138000",
  "code": "123456",    // 验证码登录时提供
  "password": "xxx"    // 密码登录时提供
}
```

> **注意**：`code` 和 `password` 二选一。验证码登录优先使用 `code`，密码登录使用 `password`。

**请求体字段**:

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `phone` | String | 是 | 手机号码 |
| `code` | String | 否 | 验证码 |
| `password` | String | 否 | 密码 |

**响应头**（成功时）:

```
authorization: Bearer eyJhbGciOiJSUzI1NiJ9...
Set-Cookie: refresh_token=...; HttpOnly; Secure; SameSite=None; Path=/
```

**响应体**: `Result`（成功时 data 为 null）

> 前端从 `Authorization` 响应头获取 Access Token，Refresh Token 由 Set-Cookie 自动管理。

#### 5.1.3 获取当前用户信息

```
GET /user/me
```

**响应**:

```json
{
  "success": true,
  "data": {
    "id": 1,
    "nickName": "user_a1b2c3",
    "icon": ""
  }
}
```

> 需要登录。用户名为 "user_" + 随机字符串。
> `nickName` 和 `icon` 从 `tb_user_info` 表获取（已从 `tb_user` 迁移至此），通过 Redis+Caffeine 缓存加速。

#### 5.1.4 根据ID查询用户

```

```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | Long | 是 | 用户ID |

#### 5.1.8 登出

```

```

> 需要登录。登出后后端会删除 Redis 中的 Token 和 Version 记录，原 Token 立即失效。

**响应**: `Result`（成功时 data 为 null）

#### 5.1.9 修改密码

```
PUT /user/password
Content-Type: application/json
Authorization: <token>

{
  "oldPassword": "Abc12345",
  "newPassword": "Xyz67890"
}
```

> 需要登录。依赖 RefreshTokenInterceptor 前置鉴权。

**密码要求**：至少 8 位，含大写字母 + 小写字母 + 数字。

**响应头**（成功时）:
```
authorization: Bearer <新 Access Token>      ← 旧 Token 已作废
Set-Cookie: refresh_token=<新 Refresh Token>; HttpOnly; Secure; SameSite=None; Path=/
```

> 修改密码后 version 自动 bump，前端需用响应头中的新 Access Token 替换本地存储的旧 Token。Refresh Token 由 Set-Cookie 自动更新。

#### 5.1.9.1 重置密码（免旧密码）

```
PUT /user/password/reset
Content-Type: application/x-www-form-urlencoded

phone=13800000000&code=123456&newPassword=Test1234
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `phone` | String | 是 | 手机号 |
| `code` | String | 是 | 短信验证码（先调 `POST /user/code` 获取） |
| `newPassword` | String | 是 | 新密码（至少 8 位，含大写 + 小写 + 数字） |

> 不需要登录，不需要旧密码。用短信验证码验证身份。
>
> 常见场景：用户忘记密码 → 发验证码 → 重置。

**响应**: `Result`

```json
{
  "success": true
}
```

#### 5.1.10 编辑个人资料

```
PUT /user/profile
Content-Type: multipart/form-data
Authorization: <token>

# 文本字段（可选）
nickName: 新昵称
city: 上海
introduce: 新的个人简介
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `iconFile` | MultipartFile | 否 | 头像图片文件（仅限 jpg/png/gif/webp，≤2MB） |
| `nickName` | String | 否 | 用户昵称（不传则不修改） |
| `city` | String | 否 | 所在城市（不传则不修改） |
| `introduce` | String | 否 | 个人简介（不传则不修改） |

> **说明**：
> - 所有字段均为可选，只传需要修改的字段即可
> - `nickName` 不允许为空字符串或纯空白
> - 头像文件上传至 `icons/` 目录（与博客图片共用 `FileService`，dev 走本地，prod 走 OSS）
> - 更换头像时后端自动删除旧头像文件
> - 前端可用 `FileReader` 实现上传前本地预览
> - 更新成功后，`GET /user/me` 和 `GET /user/{id}` 将立即返回新数据

**响应**: `Result`

```json
{
  "success": true
}
```

> - 传了 `x`/`y` → **按距离排序**（`sortBy` 无效，GEO 优先）
> - 传了 `sortBy` 但没传 `x`/`y` → **按指定字段降序排列**
> - `distance` 字段仅在使用地理坐标查询时不为 null

#### 5.8.3 存量老数据兼容

项目中存量图片路径为相对路径（如 `/imgs/blogs/...`），后端不负责 URL 转换，前端需统一处理。建议在项目中添加工具函数：

```typescript
// utils/image.ts — 图片地址兼容函数
export function getImageUrl(path: string | null | undefined): string {
    if (!path) return '';
    if (path.startsWith('http')) return path;      // 新数据：完整 URL
    return `http://localhost:8082${path}`;           // 存量老数据：拼接域名
}
```

**需要调用的字段**（必须全部覆盖）：

| 字段 | 示例值 | 调用位置 |
|------|--------|---------|
| `user.icon` | `/imgs/icons/user5-icon.png` | 用户头像 |
| `blog.icon` | `/imgs/icons/user5-icon.png` | 博客列表/详情中的博主头像 |
| `blog.images` | `/imgs/blogs/7/14/4771fefb-...jpg` | 博客中的图片展示 |
| `shop.images` | （逗号分隔的多张图） | 商铺详情页 |
| `shopType.icon` | （图片路径） | 商铺类型图标 |

> ⚠️ 若数据以逗号分隔多张图，需先 `.split(',')` 再对每张图调用 `getImageUrl()`。

## 7. 关键业务逻辑说明

### 7.1 商铺列表按距离排序

调用 `GET /shop/of/type` 时，如果传入 `x`（经度）和 `y`（纬度），后端会基于 Redis GEO 数据结构按距离从近到远排序返回，结果中 `distance` 字段表示距离。

### 7.2 笔记 Feed 流（关注者动态）

- 用户发布笔记时，系统会将笔记 ID 推送给其所有粉丝的 **Redis ZSet**（key: `feed:{userId}`，score: 当前时间戳）
- 粉丝查看动态时，通过 `GET /blog/of/follow` 实现**滚动分页**
- 滚动分页参数 `lastId` 和 `offset` 用于处理同一毫秒多条数据的翻页问题

### 7.3 秒杀流程（V2 - MQ 异步模式）

```
用户请求 POST /voucher-order/seckill/{voucherId}
          │
          ▼
  Redis Lua 脚本原子执行：
    1. 检查库存是否充足 (key: seckill:stock:{voucherId})
    2. 检查用户是否已下单 (key: seckill:order:{voucherId})
    3. 扣减库存
    4. 记录用户已下单
          │
      ┌───┴───┐
    成功      失败
      │        └→ 返回错误信息
      ▼
  生成订单ID（雪花算法）
      │
      ▼
  发送到 RabbitMQ (queue: voucher_order)
      │
      ▼
  异步消费消息 → 写入 MySQL tb_voucher_order
      │
      ▼
  返回成功
```

### 7.4 点赞系统

- 点赞数据存储在 **Redis ZSet**（key: `blog:liked:{blogId}`，member: userId，score: 点赞时间戳）
- 查询 Top 5 点赞用户时，按点赞时间从早到晚取前 5 个
- 数据库中的 `liked` 字段实时同步更新

### 7.5 用户签到

- 使用 Redis BitMap，key: `sign:{userId}`，按月存储
- offset 为当月第几天（从 0 开始）

## 8. 附录：HTTP 状态码说明

| 状态码 | 说明 | 处理方式 |
|--------|------|---------|
| `200` | 请求成功 | 正常解析 `Result` 对象 |
| `401` | 未授权 / Token 无效或过期 | 清除本地 Token，跳转登录页 |

## 9. 图片上传前端交互注意事项

> 以下问题来自 2026-07 审查，建议在实现时优先考虑。

| # | 问题 | 级别 | 建议 |
|:-:|------|:----:|------|
| F1 | 无"取消发布/放弃草稿"功能，用户无法主动清理已上传的无效图片 | 🟢 P3 | 增加草稿管理 + 过期清理 |
| F2 | 上传失败无重试策略 | 🟢 P3 | 上传组件增加重试按钮 / 自动重试 |
| F3 | 发布中未禁用提交按钮，可能重复请求导致数据错乱 | 🟢 P3 | 提交后禁用按钮直到返回 |
| F4 | 图片选择器不支持多选，逐张选择效率低 | 🟢 P3 | 支持多选图片（不超过 9 张），逐张上传并显示进度 |

### 9.1 图片选择器 UI 建议

- **多选**：支持一次选择多张图片（不超过 9 张）
- **逐张上传**：循环调用 `POST /upload/blog?blogId={blogId}` 逐张上传，前端维护上传进度条
- **失败重试**：单张上传失败可点击重试，不影响其他图片
- **删除已传**：已上传的图片可在列表中预览并单独删除（调用 `DELETE /upload/blog/delete?url={url}` 接口）
- **禁用提交**：上传过程中禁用"发布"按钮，防止重复请求
