## 亮点一：多级缓存架构 + 缓存安全三防

### 一句话概括（开场用）

> 我们这个项目用到了**Caffeine本地缓存 + Redis分布式缓存 + MySQL**三级缓存架构，同时完整处理了缓存穿透、击穿、雪崩这三个经典问题。

### 展开讲解（3~4分钟）

我们这个项目是一个类大众点评的平台，店铺信息和用户信息是访问最频繁的数据。如果每次都查数据库，在高并发场景下数据库肯定扛不住。所以我们设计了一套多级缓存体系。

**第一层是 Caffeine 本地缓存。** Caffeine 是 Google Guava Cache 的升级版，性能极高，读取只需要不到 0.1 毫秒。我们把用户信息放在本地缓存里，配置了最大 10000 条，写入后 20 分钟过期，但 10 分钟就会异步刷新一次——这样热点用户的信息几乎永远在本地缓存中，根本不需要网络请求。

**第二层是 Redis 分布式缓存。** 存放店铺信息、博客内容、商铺类型列表这些所有用户共享的数据。不同数据类型设置了不同的过期时间，比如店铺缓存是 30 分钟，博客缓存是 30 到 60 分钟随机。

**第三层才是 MySQL 数据库。** 争取绝大多数请求在第一层或第二层就返回了，打到数据库的请求非常少。

但有了缓存还不够，还需要处理三个经典问题。

**第一个是缓存穿透。** 就是恶意用户故意请求一个肯定不存在的 key，比如查询 ID 为 -1 的店铺，缓存肯定没有，请求直接打到数据库。如果有人用这种手段高频攻击，数据库很容易被打挂。我们的做法是：**如果数据库也查不到，就在 Redis 里写入一个空值，设置 2 到 5 分钟的过期时间。** 下次同样的 key 再来，Redis 直接返回空，不会再查数据库。这就是用空值缓存来防穿透。

**第二个是缓存击穿。** 某个热点 key 在过期的那一瞬间，大量并发请求同时发现缓存过期了，全部打到数据库。我们的解决方案有两个，我都实现了。第一个是**互斥锁方案**：当发现缓存过期后，先尝试获取 Redis 分布式锁，拿到锁的线程去查数据库写缓存，没拿到锁的线程 sleep 一小会儿然后重试。第二个是**逻辑过期方案**：缓存里存的不只是数据，还有一个过期时间标记。请求来了，如果发现逻辑时间还没过期，直接返回缓存数据；如果发现过期了，就起一个后台线程异步去重建缓存，当前请求**直接返回旧数据**，不等待。这个方案的好处是**完全不阻塞**，用户体验更好。

> **面试官追问方向：** 为什么有互斥锁还要做逻辑过期？
> **回答思路：** 互斥锁方案中，没拿到锁的线程要 sleep 重试，在高并发场景下会增加响应延迟。逻辑过期方案的核心思想是「用短暂的数据不一致换取高可用」——用户看到的数据最多延迟一两秒，但不会卡住。这在非金融场景下完全可以接受。所以项目中默认使用的是逻辑过期方案，并且在 CacheClient 里封装成了泛型方法，任何数据类型都可以复用。

**第三个是缓存雪崩。** 大量 key 同时过期，导致请求全部打到数据库。我们的做法很简单：**过期时间加一个随机偏移量**，比如博客缓存的 TTL 是 30 分钟加上 0 到 30 分钟的随机值，这样 key 的过期时间就分散开了，不会集中失效。

> **面试官追问方向：** Caffeine 和 Redis 各承担什么角色？为什么不只用 Redis？
> **回答思路：** Redis 是网络请求，一次读取大约 1 毫秒。Caffeine 是进程内缓存，不到 0.01 毫秒，而且没有网络开销。对于用户信息这种高频读取、每个请求都要校验的数据，放在 Caffeine 里可以极大降低延迟。另一个关键场景是 Token 版本校验——我们第二个亮点会详细讲。简单说，Redis 是分布式共享的，Caffeine 是每个节点私有的超高速缓存，两者互补。

## 亮点二：JWT 双 Token 自动续期 + 多级版本校验

### 一句话概括（开场用）

> 我们使用了 Access Token 加 Refresh Token 的双 Token 机制，Access Token 过期后能自动无感刷新，而且通过 Caffeine 本地缓存加 Redis 两级版本校验实现秒级失效。

### 展开讲解（2~3分钟）

传统的 JWT 方案有个痛点：如果你把过期时间设短了，用户隔一会儿就要重新登录，体验很差；设长了呢，token 泄露了风险大。我们的方案是双 Token 来解决这个矛盾。

**Access Token 是 JWT 格式，有效期 30 分钟**，放在请求头的 authorization 字段里，每次请求都带上。它里面包含 userId 和一个版本号 version。**Refresh Token 是 UUID 字符串，有效期 7 天**，放在 HttpOnly Cookie 里，这样 JavaScript 读不到，可以有效防止 XSS 攻击窃取。

那 token 过期了怎么办呢？我们通过拦截器自动完成刷新，用户完全无感知。

具体流程是这样的：每次请求进来，**RefreshTokenInterceptor** 先解析 Access Token。如果 JWT 解析成功，检查一下是否快过期了——如果剩余时间在 5 到 10 分钟之间，就自动签发一个新的 Access Token 通过响应头返回给前端，前端收到后自动更新，这个过程叫**临期刷新**，版本号不变，开销很小。

如果 JWT 已经过期了，就用 Refresh Token 去换一组新的。这时版本号会升级，旧的 Access Token 立即失效。**整个刷新过程通过 Lua 脚本保证原子性**，不会出现并发冲突。

**这里有一个很关键的设计：版本校验。** JWT 里携带了一个版本号，每次刷新或者改密码时版本号都会递增。我们在校验 token 时，会先查本地的 **Caffeine 版本缓存**——如果缓存的版本号比 token 里的版本号大，说明这个 token 已经被废弃了，直接拒绝，不查 Redis。如果本地缓存没有，再查 Redis 做最终校验。这样既保证了速度，又保证了准确性。

> **面试官追问方向：** 为什么要版本号？JWT 本身不就是防篡改的吗？
> **回答思路：** JWT 防篡改但不防「失效」。假设用户修改密码，或者管理员把某个用户踢下线，JWT 本身在过期前仍然是有效的。版本号的作用是让服务端有能力主动让某个 token 失效。比如改密码时版本号 +1，所有旧版本的 token 即使 JWT 签名正确也会被拒。另一个场景是 Refresh Token 重用检测——如果有人盗用了 Refresh Token，和正常的用户同时使用，Lua 脚本检测到版本冲突，会立即升级版本号，让盗用者的 token 失效。

> **面试官追问方向：** Refresh Token 为什么要放 HttpOnly Cookie？
> **回答思路：** 安全考虑。如果放 localStorage 里，JS 能读到，一旦有 XSS 漏洞 Refresh Token 就泄露了。HttpOnly Cookie 设置了 JS 不可读，即使有 XSS 也只能窃取到 Access Token，而 Access Token 只有 30 分钟有效期，攻击窗口小很多。Access Token 放在请求头里还有一个好处，就是可以配合前端在响应头里检测到 `X-Token-Refresh: ok`，然后更新本地的 Access Token。

## 亮点三：Redis GEO 地理检索 + 附近店铺排序

### 一句话概括（开场用）

> 我们用 Redis 的 GEO 数据结构实现了附近店铺搜索功能，支持按距离排序和多维度排序，并且在排序字段上做了防 SQL 注入的保护。

### 展开讲解（1.5~2分钟）

大众点评这类平台有一个核心功能就是找附近的店铺。我们这一块的实现用到了 Redis 的 GEO 特性。

店铺在创建的时候，它的经纬度信息会存入 Redis GEO，按店铺类型分组，key 的格式是 `shop:geo:{typeId}`。查询时传入用户当前的经纬度和范围，比如 5 公里，Redis 直接返回范围内所有店铺的 ID 和距离，而且这些结果默认就是按距离排好序的。

然后我们拿着这些 ID，去 MySQL 里把店铺的完整信息查出来，用 `ORDER BY FIELD(id, ...)` 保持和 GEO 返回的顺序一致。

这里有一个亮点是**双模式设计**。我们查询店铺的接口 `/shop/of/type`，如果前端传了经纬度就走 GEO 附近查询，如果没传就走普通的分页查询，支持按评分、销量、评论数、均价等字段排序。

项目中的另一个亮点是**排序字段用了白名单机制**——我们在代码里定义了一个 `SORTABLE_FIELDS` 的 Map，只有白名单里的字段才能拼到 SQL 里。这不是直接用前端传的字段名拼 SQL，可以防止 SQL 注入。

> **面试官追问方向：** 为什么用 Redis GEO 而不是 MySQL 的空间索引？
> **回答思路：** Redis GEO 的数据结构底层是跳表加 geohash，写入和查询的延迟非常稳定，大约 1 毫秒。MySQL 的空间索引（SPATIAL INDEX）也可以实现类似功能，但在高并发场景下，每次查询都打到数据库的开销比较大。用 Redis 做这件事相当于把地理查询也缓存化了，而且 GEO 的计算是在 Redis 内部完成的，不用把经纬度拉到应用层再算距离。当然，Redis GEO 也有局限性，比如范围较大时精度会下降，我们设的 5 公里是在精度和覆盖范围之间的一个合理折中。

## 亮点四：分布式点赞系统 —— Set + ZSet 组合防刷

### 一句话概括（开场用）

> 点赞功能我们用 Redis Set 来保证一人一赞，用 ZSet 来记录点赞顺序，再用分布式锁防止并发操作导致点赞数不准。

### 展开讲解（1.5~2分钟）

博客的点赞功能看起来简单，但并发场景下容易出问题。比如用户快速点击点赞和取消，如果没有保护措施，数据库里的点赞数就乱掉了。

**我们的设计用了两个 Redis 数据结构配合。** 第一个是 Set，key 是 `user:liked:{userId}`，里面存的是这个用户点赞过的所有博客 ID。每次点赞前先查这个 Set，如果已经点过了就取消点赞，没点过就新增。这就保证了**一人一赞**。

第二个是 ZSet，key 是 `blog:liked:{blogId}`，里面存的是给这篇博客点赞的用户 ID，score 是当前时间戳。作用是**记录谁点了赞以及点赞的顺序**，我们查询点赞列表时只需要取 ZSet 的前 5 名，就是「谁赞了我」的功能。

**最关键的是分布式锁**。点赞操作涉及三步：查 Set、更新 MySQL 的 liked 字段、更新 Set 和 ZSet。这三步不是原子的，高并发下如果两个请求同时进来，可能导致点赞计数错乱。我们加了 `lock:like:{blogId}:{userId}` 这把锁，同一个用户同一篇博客在 3 秒内只能有一个点赞操作在执行，防止了并发问题。

点赞完成后还会同步刷新 Redis 里博客的缓存数据，确保其他用户看到的点赞数是实时的。

> **面试官追问方向：** 为什么用 Set + ZSet 两个结构，一个 Set 不够吗？
> **回答思路：** Set 只能判断「有没有点赞」，但不知道「谁先点的」。ZSet 的 score 是时间戳，可以按时间排序。比如博客详情页要展示「谁赞了我」的前 5 个头像，就需要 ZSet 来取按时间排序的前 5 个用户。如果只用 Set，想实现这个排序效果就得额外查数据库做排序，性能就差了。所以我们的使用方式是：Set 用于一人一赞的判定，ZSet 用于点赞列表的展示，各司其职。
