## Phase 0 — 紧急修复（安全漏洞）

### 0.1 验证码原子消费（ConsumeVerifyCode.lua）

**问题**：`login()` 中先用 GET 取验证码，再手动比对，没有删除操作。同一验证码可多次使用，存在重放攻击风险。

**方案**：新建 `ConsumeVerifyCode.lua` 脚本，原子执行 GET + DEL。`AuthService.consumeVerifyCode()` 通过该脚本消费验证码，消费后立即删除，无法二次使用。

**涉及文件**：
- `resources/ConsumeVerifyCode.lua`（新建）
- `config/RedisLuaConfig.java`（注册 Bean）

### 0.2 验证码发送频率限制

**问题**：`sendCode()` 无频率限制，可被脚本无限调用，短信接口滥用。

**方案**：每次发送前检查 Redis 中是否已存在 `login:code:{phone}` key，存在则返回"发送太频繁"。

**涉及文件**：
- `service/impl/UserServiceImpl.sendCode()`

### 0.3 移除 HttpSession

**问题**：`UserController`、`IUserService`、`UserServiceImpl` 中 `sendCode()` 和 `login()` 方法签名携带 `HttpSession` 参数，但项目早已全面 Token 化，session 从未使用。

**方案**：删除所有 `HttpSession` 参数。

**涉及文件**：
- `controller/UserController.java`
- `service/IUserService.java`
- `service/impl/UserServiceImpl.java`

### 0.4 LoginSetToken.lua 去掉先 DEL

**问题**：Lua 脚本中先 DEL 再 SET 写入 Token 和 Version。主从复制场景下，DEL 可能已同步到从库但 SET 尚未落盘，主节点宕机切换后 key 丢失。

**方案**：改为直接 SET 覆盖，删除无意义的 DEL。

**涉及文件**：
- `resources/LoginSetToken.lua`

### 0.5 验证码改为纯数字

**问题**：`RandomUtil.randomString(6)` 生成字母+数字验证码，真实短信验证码应为纯数字。

**方案**：改为 `RandomUtil.randomNumbers(6)`。

**涉及文件**：
- `service/impl/UserServiceImpl.login()`（已迁移到 AuthService，login 中使用了新用户的 nickName 生成）

### 0.6 修复拼写错误（4 处）

| 位置 | 错误 → 正确 |
|------|-------------|
| `RefreshTokenInterceptor.java:165` | `remaningTime` → `remainingTime` |
| `JwtUtil.java:181` | `valiateAndGetClaimFromToken` → `validateAndGetClaimFromToken` |
| `UserServiceImpl.java:403` | `fileName = fileName;` 空赋值删除 |
| `TokenRefreshCode.java:43` | `USEINFO_NOT_FOUND` → `USERINFO_NOT_FOUND` |

### 0.7 checkTokenHeader 只校验 authorization

**问题**：`checkTokenHeader()` 同时要求 `authorization` 和 `Refresh-Token` 两个请求头，导致 Refresh-Token 过期后用户连 logout 都无法调用（请求无法到达 Controller）。

**方案**：改为只校验 `authorization`，`Refresh-Token` 仅在需要刷新时才读取。

**涉及文件**：
- `interceptor/RefreshTokenInterceptor.checkTokenHeader()`

## Phase 1 — 测试代码剥离 + Token 约定统一

### 1.1 测试代码剥离到 TokenTestUtil

**问题**：`UserServiceImpl` 中混入约 200 行测试专用代码（批量生成 Token、CSV 导出），`IUserService` 接口暴露了 `exportTokenAndRefreshTokenToCsv()` 方法。

**方案**：测试代码迁移到独立的 `TokenTestUtil` 工具类，`IUserService` 接口清理。

**涉及文件**：
- `utils/TokenTestUtil.java`（新建）
- `service/IUserService.java`（移除方法声明）
- `service/impl/UserServiceImpl.java`（删除字段和方法）
- `service/impl/TestServiceImpl.java`（改为调用 TokenTestUtil）

### 1.2 Token 统一走响应头

**问题**：登录接口通过**响应体**返回 Token（`Result.ok({ token, refreshToken })`），而拦截器刷新 Token 后通过**响应头**返回（`response.setHeader("authorization", ...)`），两种方式不一致。

**方案**：`login()` 返回 `TokenPair`，Controller 通过 `response.setHeader()` 写入响应头。Service 不再关心 HTTP 细节。

```
// 前端获取 Token 的方式
const token = response.headers.get('authorization');
const refreshToken = response.headers.get('Refresh-Token');
```

**涉及文件**：
- `controller/UserController.java`（接收 HttpServletResponse，写响应头）
- `service/IUserService.java`（login() 返回类型 Result → TokenPair）
- `service/impl/UserServiceImpl.java`（返回 TokenPair，不再组装 map）
- `config/WebExceptionAdvice.java`（新增 IllegalArgumentException 处理器）
- `md/前端开发文档.md`（同步更新）

## Phase 2 — 抽取 AuthService

### 2.1 新增 AuthService 接口 + 实现

**设计**：纯业务服务，无 HTTP 依赖。负责 Token 生命周期管理。

```
AuthService 接口方法：
├─ generateTokenPair(userId)    → TokenPair      ← 登录创建双 Token
├─ validateAccessToken(token)   → ValidationResult ← JWT 解析 + 两级版本校验
├─ refreshTokenPair(refresh)    → TokenPair      ← 刷新（待实现）
├─ revokeTokens(userId)         → void           ← 登出吊销
├─ consumeVerifyCode(phone,code)→ boolean        ← 原子消费验证码
└─ cacheUserInfo(userDTO)       → void           ← 缓存用户信息
```

**涉及文件**：
- `dto/TokenPair.java`（新建） — accessToken + refreshToken + version
- `dto/ValidationResult.java`（新建） — valid + userId + version + needsRefresh
- `service/AuthService.java`（新建）
- `service/impl/AuthServiceImpl.java`（新建）

### 2.2 UserServiceImpl.login() 简化

**问题**：原 `login()` 方法约 70 行，混入手机号校验 → 验证码消费 → 用户查/创建 → Token 生成 → Redis Lua 写入 → 用户信息缓存 → 返回 map 等全部逻辑。

**方案**：拆分为 6 步清晰流水线，验证码消费和 Token 生成委托 AuthService。

```java
① 手机号格式校验（RegexUtils）
② 原子消费验证码（AuthService.consumeVerifyCode）
③ 查用户/创建用户
④ 生成双 Token（AuthService.generateTokenPair）
⑤ 缓存用户信息（AuthService.cacheUserInfo）
⑥ 返回 TokenPair
```

**涉及文件**：
- `service/impl/UserServiceImpl.java`（login 从 ~70 行缩至 ~48 行，删除 6 个未用字段）

### 2.3 Lua 异常处理修复

**问题**：`generateTokenPair()` 中 `JSONUtil.toBean(execute, LuaResult.class)` 在 `execute` 为 null 时 NPE，且异常被吞掉。

**方案**：增加 null 判断 + code != 1 时 warn 日志。

## Phase 3 — 权限 + 缓存增强

### 3.1 /user/{id} 和 /user/info/{id} 增加权限校验

**规则**：仅当前登录用户（`UserHolder.getUserId()`）可访问自己的信息，越权返回业务错误。

```java
if (!currentUserId.equals(userId)) {
    return Result.fail("无权访问该用户信息");
}
```

**涉及文件**：
- `controller/UserController.java`

### 3.2 /user/{id} 增加缓存

**方案**：使用 `CacheClient.queryById()` 实现缓存穿透防护的读取。

```
缓存 key: "cache:user:{id}"
缓存 TTL: 30 分钟
空值保护: 2 分钟
```

**涉及文件**：
- `controller/UserController.java`

### 3.3 密码登录

**问题**：`LoginFormDTO.password` 字段存在但从未使用，新用户只设手机号不设密码，无密码登录能力。

**方案**：
1. `PasswordEncoder` 升级为 BCrypt（兼容旧 MD5+盐格式，登录成功后自动升级）
2. `login()` 二选一：传 `code` 走验证码，传 `password` 走密码登录
3. 密码登录含安全防护：连续 10 次失败锁定 30 分钟（Redis `login:lock:{phone}`）
4. 修改密码 `PUT /user/password`：校验旧密码 → BCrypt 加密新密码 → bump version → 返回新 Token

**涉及文件**：
- `pom.xml`（添加 spring-security-crypto）
- `utils/security/PasswordEncoder.java`
- `utils/RegexUtils.java`（新增 isPasswordInvalid）
- `dto/PasswordChangeDTO.java`（新建）
- `utils/redis/RedisConstants.java`（新增 3 个 key）
- `service/IUserService.java`（新增 changePassword）
- `service/impl/UserServiceImpl.java`
- `controller/UserController.java`

## Phase 4 — 拦截器瘦身

### 4.1 Token 校验委托 AuthService

**问题**：`tryRefreshIfNeeded()` 内联执行 JWT 解析 → Caffeine 版本校验 → Redis 版本比对 → 临期判断，约 50 行，与 AuthService 职责重复。

**方案**：替换为 `authService.validateAccessToken(token)`，删除 `validateLocalVersionCache()` 方法。

```
Before: jwtUtil.parse → Caffeine check → Redis check → update cache → calc expiry
After:  authService.validateAccessToken(token)  →  ValidationResult
```

**涉及文件**：
- `interceptor/RefreshTokenInterceptor.java`（删 ~30 行内联校验代码）
- `service/impl/AuthServiceImpl.java`（已有实现，无需新增）

### 4.2 刷新逻辑委托 AuthService

**问题**：`refreshDeadlineToken()` 和 `refreshExpiredToken()` 两个内联方法约 180 行，直接操作 Redis Lua 脚本，与 AuthService 职责重复。

**方案**：全部委托 `authService.refreshTokenPair(accessToken, refreshToken, userId, version, isExpired)`。拦截器从 433 行瘦身至 135 行（-70%）。

```
Before: 拦截器 → 9 个注入依赖 + 8 个私有方法 + 直接调 Lua
After:  拦截器 → 2 个注入依赖 + 3 个私有方法 → 全部委托 AuthService
```

**涉及文件**：
- `interceptor/RefreshTokenInterceptor.java`（删除 2 个刷新方法 + 6 个注入依赖）
- `service/impl/AuthServiceImpl.java`（新增 doDeadlineRefresh + doExpiredRefresh + updateLocalVersionCache）

### 4.2 拦截器无脑回写响应头删除

**问题**：`preHandle()` 中 `response.setHeader("authorization", token)` 将请求头的值原样写回响应头，对大多数请求无意义。

**方案**：删除，登录和刷新两处写响应头由各自的场景自行处理。

**涉及文件**：
- `interceptor/RefreshTokenInterceptor.java`
