## 开场（30 秒）

面试官好，我这个项目是一个**大众点评类的本地生活服务平台**，后端用的 Spring Boot + MyBatis-Plus + Redis + MySQL + Caffeine + RabbitMQ。我主要讲四个技术亮点。

---

## 亮点一：多级缓存 + 缓存三防（3 分钟）

我们这个项目设计了**三级缓存**：Caffeine 本地缓存 → Redis → MySQL。用户信息和 token 版本放 Caffeine，读取不到 0.1 毫秒；店铺和博客数据放 Redis；数据库兜底。

同时解决了三个缓存经典问题：

**穿透**——恶意请求不存在的数据。做法是：数据库查不到也在 Redis 写空值，设短 TTL。下次同样的 key 直接返回空，不打数据库。

**击穿**——热点 key 过期瞬间，高并发打到数据库。我实现了两种方案：
- 互斥锁：拿锁的线程查 DB，没拿到的 sleep 重试
- **逻辑过期（默认用这个）**：缓存里存一个过期时间标记，过期后后台线程异步重建缓存，当前请求直接返回旧数据，**完全不阻塞**。已经在 `CacheClient` 里封装成通用方法，任何数据都能复用

**雪崩**——大量 key 同时过期。做法：TTL 加随机偏移量，过期时间分散开。

---

## 亮点二：JWT 双 Token 自动续期（2 分钟）

传统 JWT 痛点：到期就得重新登录。我们的方案：

Access Token（JWT，30 分钟）放请求头，Refresh Token（UUID，7 天）放 **HttpOnly Cookie**，JS 读不到，防 XSS。

**关键设计：版本号。** 每个 token 携带一个版本号，每次刷新或改密码递增。校验流程：
1. Caffeine 本地缓存查版本 —— 不匹配直接拒，零网络开销
2. Redis 最终校验

**两种刷新策略：**
- 临期刷新（剩 5-10 分钟）：版本不变，直接换 token
- 过期刷新：版本升级，用 Lua 脚本保证原子性

改密码时版本号 +1，**所有已签发的 token 立即失效**。

## 亮点三：Redis GEO 附近店铺（1 分钟）

店铺经纬度按类型存入 Redis GEO（`shop:geo:{typeId}`）。查询时传入用户坐标和范围（5 公里），Redis 直接返回范围内店铺 ID 和距离。

接口支持**双模式**：传了经纬度就走 GEO 附近查询，没传就走普通分页，**支持按评分、销量、评论数、均价排序**。

排序字段用了**白名单机制**——只在预定义的 `SORTABLE_FIELDS` 里取值，防止 SQL 注入。

---

## 亮点四：分布式点赞（1.5 分钟）

点赞用**两个 Redis 结构配合**：

- **Set**（`user:liked:{userId}`）：记录用户点赞过的博客，保证一人一赞
- **ZSet**（`blog:liked:{blogId}`）：按时间排序的点赞列表，取 Top 5

每次点赞操作加了**分布式锁**（`lock:like:{blogId}:{userId}`），防止并发点赞导致计数不准。操作完成后同步更新 MySQL `liked` 字段和博客缓存。

## 附：如果面试官追问

**Q：逻辑过期和互斥锁怎么选？**
> 逻辑过期追求性能，数据可能短时不一致（一两秒）；互斥锁追求强一致但有阻塞。非金融场景推荐逻辑过期。

**Q：为什么用 Caffeine 不用 Redis 就够了？**
> Caffeine 是本地内存，0 网络开销。用户信息和 token 版本每个请求都查，放本地能极大降延迟。Redis 是分布式共享，适合多实例共享的数据。

**Q：点赞为什么用两个结构？**
> Set 查「有没有点赞」，ZSet 查「谁点了赞按时间排序」。各司其职，不重复查数据库。
