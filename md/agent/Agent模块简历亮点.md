# Agent 模块 — 简历项目亮点

> **用途**: 面试自我介绍用  
> **版本**: v3.0 (2026-07-24)  
> **风格**: 务实工程叙事，非 PPT 包装

---

## 项目职责与难点攻克

**LLM Agent 智能对话模块** — 在 Spring AI + DashScope 基础上，为电商系统接入自然语言交互能力，让 AI 能安全地查询/操作业务数据。

---

**难点 1：AI 调接口不失控** — 聊天场景下，AI 可能生成恶意指令（如"删库"、"批量发帖"）或进入穷举循环。我的做法是在 Agent 执行链上设**两道闸口**：

- **前置闸口（输入阶段）**：关键词 + 正则做注入检测，命中即阻断；
- **执行闸口（工具调用阶段）**：Guard 代理层统一包装 ToolCallback，过 4 个 Policy（高危名单、需确认名单、正则匹配、Redis 限流），任一拒绝即拦截；同时工具方法上的 `@RequiredDataPermission` AOP 切面做数据归属兜底校验。

两层任一异常均 **Fail-Close**，不放行。

---

**难点 2：两阶段架构 — 先规划后执行，不走弯路** — 最初设计是第一轮 AI 调用时就绑定全部工具（Spring AI 默认行为），AI 第一次回复就直接调了工具，然后规划器又规划一遍，导致工具重复执行。重构后改为**两阶段**：

- **Phase 1**：AI 纯文本回复，不注册任何工具
- **Phase 2**：如果触发词（"统计"、"分析"、"对比"等）命中，进入 TaskPlanner 做 AI 规划 → Java 校验 → 串行执行 → LLM 聚合

核心收益：工具只执行一次，AI 调用的 Token 不浪费，日志链路清晰。每轮执行都有 `[Round N] ①/②/③` 的阶段标记。

---

**难点 3：AI 规划不可靠，Java 做三层校验兜底** — 让 AI 决定调哪些工具（`askAiForPlan()` 返回 JSON 数组），但 AI 可能生成不存在的工具名、漏参数、或重复规划已完成的任务。我在 `validatePlan()` 里做了三层校验：

1. JSON 语法（parseable? isArray?）
2. 工具存在性（在 ToolBeanCollector 中注册过?）
3. 历史状态（已完成/终极失败?不及格跳过）

校验通过后**强制追加 LLM_REASON 聚合任务**，确保工具结果不被原样拼接，而是经 AI 总结后呈现。空计划也有兜底（所有工具被跳过时保持原 AI 回复），不会死循环。

---

**难点 4：异步线程丢失登录上下文** — Spring AI 的工具回调在独立线程执行，`ThreadLocal` 拿不到用户信息。解决方案很简单：发起调用时用 `.toolContext(Map.of("userId", userId))` 把用户 ID 显式传进去，工具方法加一个 `ToolContext` 参数接收，不走 `UserHolder`。麻烦在于需要确保团队写的每个新工具都遵守这个约定，不能漏传。

---

**难点 5：工具多了之后 Guard 包装容易遗漏** — 工具数量增加后，手动注册 + 手动包 Guard 流程全靠人工记，新人容易漏掉守卫导致安全漏洞。我写了一个 `@TargetTool` 注解，启动时自动扫描带此注解的 Bean，通过 `ToolCallbacks.from()` 批量注册，同时自动用 Guard 包装，让安全守卫成为不可跳过的流程。本质上只是利用了 Spring 的组件扫描机制，不是什么框架设计，但核心收益是**安全不依赖人肉记忆**。

---

**效果**：AI 调用上线后未发生工具越权操作。两阶段架构下无工具重复调用问题。
