# Agent 模块 — 简历项目亮点

> **用途**: 面试自我介绍用  
> **版本**: v2.1 (2026-07-22)  
> **风格**: 务实工程叙事，非 PPT 包装

---

## 项目职责与难点攻克

**LLM Agent 智能对话模块** — 在 Spring AI + DashScope 基础上，为电商系统接入自然语言交互能力，让 AI 能安全地查询/操作业务数据。

---

**难点 1：AI 调接口不失控** — 聊天场景下，AI 可能生成恶意指令（如"删库"、"批量发帖"）或进入穷举循环。我的做法是在 Agent 执行链上设**两道闸口**：

- **前置闸口（输入阶段）**：关键词 + 正则做注入检测，命中即阻断；
- **执行闸口（工具调用阶段）**：Guard 代理层统一包装 ToolCallback，过 4 个 Policy（高危名单、需确认名单、正则匹配、Redis 限流），任一拒绝即拦截；同时工具方法上的 `@RequiredDataPermission` AOP 切面做数据归属兜底校验。

两层任一异常均 **Fail-Close**，不放行。

---

**难点 2：异步线程丢失登录上下文** — Spring AI 的工具回调在独立线程执行，`ThreadLocal` 拿不到用户信息。解决方案很简单：发起调用时用 `.toolContext(Map.of("userId", userId))` 把用户 ID 显式传进去，工具方法加一个 `ToolContext` 参数接收，不走 `UserHolder`。麻烦在于需要确保团队写的每个新工具都遵守这个约定，不能漏传。

---

**难点 3：工具多了之后 Guard 包装容易遗漏** — 工具数量增加后，手动注册 + 手动包 Guard 流程全靠人工记，新人容易漏掉守卫导致安全漏洞。我写了一个 `@TargetTool` 注解，启动时自动扫描带此注解的 Bean，通过 `ToolCallbacks.from()` 批量注册，同时自动用 Guard 包装，让安全守卫成为不可跳过的流程。本质上只是利用了 Spring 的组件扫描机制，不是什么框架设计，但核心收益是**安全不依赖人肉记忆**。

---

**效果**：AI 调用上线后未发生工具越权操作。
